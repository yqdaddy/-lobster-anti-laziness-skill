## 一、竞品全景扫描

### 1.1 mem9 — 最热门的持久记忆外挂

| 维度 | 详情 |
|------|------|
| **定位** | AI Agent 的持久记忆层，跨会话、跨机器、跨用户 |
| **技术** | Go 服务端 + TiDB Cloud (MySQL + 向量搜索) + 混合检索 |
| **核心能力** | ① 持久记忆 ② 多 Agent 共享 ③ 混合搜索(向量+关键词) ④ 可视化 Dashboard |
| **集成** | OpenClaw, Hermes Agent, Claude Code, OpenCode, Codex, Dify |
| **规模** | 企业级，处理 10K+ 记忆条目 |
| **成本** | 托管服务 / 自托管需数据库基础设施 |

**mem9 的杀手锏**: 
- "记忆质量评估" — 把"记忆感觉不对"变成可诊断的指标
- 向量+SQL 混合 — 既能语义搜索又能精确过滤
- 跨机器同步 — 换电脑记忆不丢

**mem9 的弱点**:
- 需要基础设施 (TiDB Cloud)
- 不透明 — 记忆存在数据库里，用户无法直接阅读
- 重 — 适合企业，个人用户过度设计

---

### 1.2 OpenViking — 字节跳动的上下文数据库

| 维度 | 详情 |
|------|------|
| **定位** | Agent 专用上下文数据库，替代传统 RAG |
| **技术** | 文件系统范式 + L0/L1/L2 三层上下文 + 目录递归检索 |
| **核心能力** | ① Token 消耗砍 91% ② 任务完成率提升 43% ③ 可视化检索轨迹 ④ 自动会话管理 |
| **集成** | OpenClaw, OpenCode, Claude Code, LangChain |
| **规模** | 大型项目，支持 VLM + Embedding 多模型 |
| **成本** | 需 Python + Rust 环境，配置复杂 |

**OpenViking 的杀手锏**:
- **L0/L1/L2 分层** — 抽象/概览/详情三层，按需加载省 Token
- **文件系统范式** — `viking://` URI，像管理文件一样管理记忆
- **可视化检索轨迹** — 黑盒变透明，能 debug 检索逻辑
- **效果数据** — 有实测：Token 省 91%，完成率提 43%

**OpenViking 的弱点**:
- 重 — 需要 VLM + Embedding 模型 + 配置复杂
- 学习曲线陡 — 文件系统概念需要适应
- 个人用户杀鸡用牛刀

---

### 1.3 Memoh-v2 — "超龙虾"全功能平台

| 维度 | 详情 |
|------|------|
| **定位** | 比 OpenClaw 强大数倍的人性化 Agent 平台 |
| **技术** | Docker 容器隔离 + Qdrant 向量 + BM25 关键词 + LLM 提取 |
| **核心能力** | ① 真隔离(每 Bot 一个容器) ② 三层记忆 ③ 自我进化 ④ 多 Bot 协作 ⑤ 21 项技能 |
| **集成** | Telegram/飞书/微信/Discord/Web/CLI |
| **规模** | 全功能平台，347 Stars |
| **成本** | 零基础设施成本，仅需 LLM API 费用 |

**Memoh-v2 的杀手锏**:
- **真隔离** — 每个 Bot 独立 containerd 容器，文件系统/网络隔离
- **三层记忆** — Qdrant(语义) + BM25(关键词) + LLM(智能提取)
- **时间衰减** — 30 天半衰期自动降权，新记忆权重高
- **记忆压缩** — LLM 自动合并冗余、去除矛盾
- **解决方案记忆** — 自动提取「问题→方案」对，下次直接召回
- **零成本** — SearXNG + Qdrant + BM25 + Containerd 全自托管

**Memoh-v2 的弱点**:
- 重 — 需要 Docker，全套基础设施
- 平台锁定 — 是完整平台而非可插拔组件
- 个人用户可能不需要这么多功能

---

### 1.4 GBrain — YC 总裁的个人 AI 记忆库

| 维度 | 详情 |
|------|------|
| **定位** | 个人 AI 知识管理系统，管理过万条 Markdown |
| **技术** | Markdown + Git + PostgreSQL + pgvector |
| **核心能力** | ① 混合搜索(精确+语义) ② 多 Agent 共享 ③ 数据导入(Gmail/日历/X) ④ 语音输入 |
| **规模** | 7.1k+ Stars，管理 10,000+ Markdown + 3,000 人物页 |
| **成本** | 本地 PGLite 秒级初始化，或 Postgres 自托管 |

**GBrain 的杀手锏**:
- **Git + Postgres 混合** — Markdown 人类可读 + 数据库高性能搜索
- **多 Agent 共享** — "git+postgres works wonderfully with multiple agents"
- **数据导入** — Gmail、日历、X、会议转录、语音全接入
- **实战验证** — Garry Tan 自己管理 10k+ 文件， battle-tested

**GBrain 的弱点**:
- 需要 Postgres — 虽然轻量但仍需数据库
- 偏向知识管理 — 不是专门的 Agent 执行纪律框架
- 无防偷懒机制 — 没有执行层面的约束

## 三、龙虾超记忆防偷懒技能 的差异化价值

### 3.1 我们独有的（别人没有）

1. **🦞 防偷懒框架** — 业界首个系统性 AI 偷懒检测 + 六问法自检
2. **🛡️ 用户否决硬规则** — 解决"自检悖论"的外部防线
3. **⚡ 零依赖** — 唯一真正零安装、零配置的方案
4. **📂 搬家而非压缩** — 冷记忆归档，信息零丢失
5. **🔍 完全透明** — 每行记忆都可读、可编辑、可审计

### 3.2 我们不足的（别人有）

1. **❌ 无混合搜索** — 只有文本搜索，无向量语义检索
2. **❌ 无跨机器同步** — 本地文件，换电脑记忆丢失
3. **❌ 无可视化** — 纯文本，无 Dashboard
4. **❌ 无自动压缩** — 需手动归档，无 LLM 自动合并
5. **❌ 无时间衰减** — TTL 是二元过期，无平滑降权
6. **❌ 无多 Agent 共享** — 单用户设计
7. **❌ Token 未优化** — 读取整文件，无智能裁剪

## 四、v4.0 优化方案："轻量级混合记忆"

**核心思路**: 保持零依赖优势，通过可选插件层吸收竞品精华。不做重基础设施，做"可升级的轻量框架"。

### 4.1 新增：分层记忆加载（借鉴 OpenViking）

**问题**: 每次读取整份 MEMORY.md，Token 浪费。

**方案**: 引入 L0/L1/L2 分层，但用 Markdown 实现：

```markdown
# MEMORY.md

## ⚡ L0 抽象（~100 tokens）
身份+3个待办+2个项目+铁律速记

## 📋 L1 概览（~2k tokens）
用户信息+环境+待办列表+项目状态

## 📚 L2 详情（按需加载）
踩坑经验+决策记录+审计日志
```

**实现**:
- 不改变文件格式 — 只是约定分层
- AI 启动只读 L0+L1（2-3k tokens）
- L2 按需读取（遇到相关任务时）
- 归档文件作为 L3（历史冷数据）

**效果**: Token 消耗减少 60-80%，同时保持 Markdown 透明性。

---

### 4.2 新增：可选向量搜索层（借鉴 mem9/GBrain）

**问题**: 1000+ 条目后文本搜索不够。

**方案**: 提供**可选**的轻量级向量搜索插件，不强制：

```
lobster-anti-laziness-skill/
├── core/                    # 零依赖核心（保留）
└── plugins/
    └── vector-search/       # 可选插件
        ├── README.md
        ├── setup.sh         # 一键安装：pip install chromadb
        └── sync.py          # Markdown ↔ 向量库同步脚本
```

**实现**:
- 核心保持零依赖
- 用户可选安装 `chroma` 或 `qdrant-client`
- 提供同步脚本：Markdown 变更 → 自动更新向量索引
- 搜索时：先文本搜索，无结果时回退到向量搜索
- 完全可选 — 不装插件也能正常使用

**对标**: mem9 的混合搜索，但以插件形式提供，不破坏零依赖承诺。

---

### 4.3 新增：时间衰减机制（借鉴 Memoh-v2）

**问题**: TTL 是二元过期（到期就标为过期），不够平滑。

**方案**: 引入半衰期权重：

```markdown
- **[fact]** API base URL is https://api.example.com/v2
  `TTL: permanent` `weight: 1.0` (confirmed 2026-05-13)

- **[mistake]** ImportError on module X → Solution: install Y
  `TTL: 90 days` `weight: 0.7` `halflife: 30 days` (2026-05-01)
  # 30天后权重降到 0.35，60天后 0.175，逐渐淡出
```

**实现**:
- 纯 Markdown 标记，无依赖
- AI 召回时考虑权重：高权重优先，低权重靠后
- 用户审计时，低权重条目优先提示"是否删除"
- 不自动删除 — 只是降低优先级

---

### 4.4 新增：记忆压缩助手（借鉴 Memoh-v2）

**问题**: 搬家是手动操作，需要 AI 判断什么该搬。

**方案**: 提供压缩助手脚本（可选）：

```bash
# 分析 MEMORY.md，建议压缩方案
python plugins/compression-helper.py MEMORY.md

# 输出：
# "发现 5 条相似踩坑经验，建议合并为 1 条"
# "发现 3 个已完成项目，建议归档"
# "发现 2 条 [unverified] 超过 30 天，建议删除"
```

**实现**:
- 纯本地脚本，无外部依赖
- 使用简单文本相似度（Jaccard）
- 只提供建议，不自动执行 — 用户确认后才修改
- 符合"用户拥有最终解释权"原则

---

### 4.5 新增：跨机器同步方案（借鉴 mem9/GBrain）

**问题**: 本地文件，换电脑记忆丢失。

**方案**: 提供多种同步选项，不强制：

```markdown
## 同步选项（任选其一）

### 选项 A: Git 同步（推荐）
```bash
git init
git add SOUL.md MEMORY.md memory/
git remote add origin https://github.com/yourname/lobster-anti-laziness-skill.git
git push
```

### 选项 B: Syncthing / Dropbox
直接同步文件夹。

### 选项 C: 可选的 mem9 集成
安装 mem9 插件，将热记忆同步到 mem9 云端。
```

**实现**:
- 核心不依赖任何同步机制
- 文档提供详细指南
- 高级用户可选 mem9 集成插件

---

### 4.6 增强：可视化审计（新功能）

**问题**: 纯文本审计不够直观。

**方案**: 提供简单的 HTML 报告生成器：

```bash
python plugins/audit-report.py

# 生成 audit-report.html：
# - 记忆健康度仪表盘
# - [unverified]/[user-vetoed] 分布图
# - TTL 到期时间线
# - 记忆增长趋势
```

**实现**:
- 纯 Python 标准库生成 HTML（无前端依赖）
- 可选打开浏览器查看
- 不强制 — 纯文本审计依然可用

---

### 4.7 保留并强化：防偷懒 + 用户审计

**这是 龙虾超记忆防偷懒技能 的核心差异化，继续强化**：

```markdown
## v4.0 新增偷懒类型

| 类型 | 表现 | 检测方式 |
|------|------|----------|
| 7. **缓存型** | 用旧记忆回答新问题，不更新 | 检查记忆时间戳 vs 问题时效性 |
| 8. **过度型** | 写入过多无关记忆，污染记忆池 | 检查提取置信度，低置信度标记 |

## v4.0 增强审计
- 审计报告自动生成（HTML）
- 记忆健康度评分（0-100）
- 错误记忆趋势分析
```

## 五、v4.0 架构演进

```
v3.2 (当前):
┌─────────────────┐
│   Markdown 文件  │  ← 唯一存储
│   + 执行协议     │
└─────────────────┘

v4.0 (目标):
┌─────────────────────────────────────────┐
│           Core Layer (零依赖)            │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ L0/L1/L2│ │ 防偷懒  │ │ 用户审计│   │
│  │ 分层加载│ │ 六问法  │ │ 硬规则  │   │
│  └─────────┘ └─────────┘ └─────────┘   │
└─────────────────────────────────────────┘
                    ↑
┌─────────────────────────────────────────┐
│        Optional Plugin Layer (可选)      │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │向量搜索  │ │时间衰减  │ │压缩助手  │   │
│  │(chroma)  │ │(纯标记)  │ │(本地脚本)│   │
│  └─────────┘ └─────────┘ └─────────┘   │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │审计报告  │ │跨机同步  │ │mem9集成  │   │
│  │(HTML)    │ │(Git/云)  │ │(插件)    │   │
│  └─────────┘ └─────────┘ └─────────┘   │
└─────────────────────────────────────────┘
```

**关键原则**:
- 核心层永远零依赖
- 插件层完全可选，不装不影响使用
- 所有插件都是本地脚本，无 SaaS 锁定
- 保持 Markdown 透明性 — 即使装了插件，底层仍是可读文件
